# 软件架构文档

## 1. 架构概述

本应用采用**分层架构**，结合**前后端分离**的设计模式。前端使用React构建用户界面，后端使用Rust处理业务逻辑和系统交互，通过Tauri的IPC机制进行通信。

## 3. 分层架构详细说明

**职责**:

#### 3.1.2 状态管理 (`src/stores/`)

## 4. 数据流

### 4.1 用户操作流程

```
用户操作 → React组件 → Zustand Store → IPC调用 → Rust命令 → 服务层 → 数据库
                                                              ↓
用户界面 ← React组件 ← Zustand Store ← IPC响应 ← Rust命令 ← 服务层 ← 数据库
```

### 4.2 时间追踪流程

```
用户点击"开始" 
  → timerStore.startTimer(projectId)
  → IPC: start_timer(projectId)
  → Rust: timer_service.start(projectId)
  → 数据库: 创建time_record记录
  → 前端: 定时器开始，UI更新
```

### 4.3 喝水动画流程

```
时间流逝 
  → Rust: water_animation_service.update_evaporation()
  → 数据库: 更新water_state表
  → IPC: update_water_state(state)
  → 前端: waterAnimationStore更新
  → React组件: 重新渲染动画
```

## 5. 模块间通信

### 5.1 前后端通信（IPC）

**通信协议**: Tauri IPC

**请求格式**:
```typescript
// 前端调用
invoke('command_name', { param1: value1, param2: value2 })

// 后端处理
#[tauri::command]
fn command_name(param1: Type1, param2: Type2) -> Result<ReturnType> {
    // 处理逻辑
}
```

**主要命令**:
- `get_projects`: 获取工作项列表
- `create_project`: 创建工作项
- `start_timer`: 开始计时
- `stop_timer`: 停止计时
- `get_statistics`: 获取统计数据
- `update_water_state`: 更新喝水状态
- `drink_water`: 喝水操作

### 5.2 前端状态管理

**状态管理方案**: Zustand

**Store结构**:
```typescript
interface AppState {
  projects: Project[]
  currentTimer: TimerState | null
  waterState: WaterState
  settings: Settings
  // ...
}
```

**状态更新流程**:
1. 用户操作触发Action
2. Action调用IPC命令
3. IPC响应更新Store
4. Store更新触发组件重渲染

## 6. 关键设计决策

### 6.1 为什么选择分层架构？

- **职责分离**: 各层职责清晰，易于维护
- **可测试性**: 每层可独立测试
- **可扩展性**: 易于添加新功能
- **可复用性**: 服务层可在不同场景复用

### 6.2 为什么前端使用React？

- **组件化**: 代码结构清晰，易于维护
- **生态丰富**: 丰富的组件库和工具
- **性能优秀**: React 18的并发特性
- **类型安全**: TypeScript提供类型检查

### 6.3 为什么后端使用Rust？

- **性能**: Rust性能优异，适合长期运行的应用
- **内存安全**: 无需担心内存泄漏
- **跨平台**: Rust跨平台支持良好
- **系统交互**: 方便与系统API交互

### 6.4 为什么使用SQLite？

- **轻量化**: 无需单独数据库服务器
- **本地存储**: 数据本地存储，保护隐私
- **性能**: 对于单用户应用性能足够
- **简单**: 易于部署和维护

## 8. 性能优化策略

### 8.1 前端优化

- **代码分割**: 按需加载代码
- **虚拟列表**: 长列表使用虚拟滚动
- **Memo优化**: 使用React.memo避免不必要的重渲染
- **动画优化**: 使用GPU加速，避免频繁重绘

### 8.2 后端优化

- **数据库索引**: 为常用查询字段添加索引
- **连接池**: 复用数据库连接
- **异步处理**: 耗时操作使用异步处理
- **缓存策略**: 缓存常用数据

### 8.3 内存优化

- **资源清理**: 及时清理不再使用的资源
- **懒加载**: 按需加载资源
- **数据压缩**: 压缩存储的数据

## 9. 安全性考虑

### 9.1 数据安全

- **本地存储**: 所有数据本地存储
- **数据加密**: 敏感数据可选择性加密
- **备份机制**: 定期备份数据

### 9.2 代码安全

- **输入验证**: 验证所有用户输入
- **错误处理**: 完善的错误处理，避免信息泄露
- **依赖检查**: 定期检查依赖的安全漏洞

## 10. 总结

本架构采用**分层架构 + 前后端分离**的设计模式，清晰的分层结构使得代码易于维护和扩展。前端使用React构建用户界面，后端使用Rust处理业务逻辑，通过Tauri IPC进行通信。架构设计考虑了未来功能的扩展，支持AI、皮肤等功能的无缝集成。
